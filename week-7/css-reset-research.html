Research & Make CSS Normalize and CSS Reset

Why reset at all? (What problem does it solve?)

Resets basically wipe out all the weird default styles every browser adds on its own. Back in the day, browsers all had super different paddings, margins, line-heights, and random styling on things like lists or headings. It made designs look different in every browser unless you manually fixed all of that. A reset forces everything to start from the same baseline.

Why Josh’s version over Eric’s? (2–3 differences)

  • Josh’s reset is more “modern-browser-aware,” so he skips a lot of the blanket zero-everything approach from 2007. He focuses on actual pain points we still deal with now.
  • Josh keeps useful defaults like headings staying bold, instead of wiping everything like Eric’s reset did.
  • Josh adds modern fixes (like text-wrap: balance; and smoothing for media and buttons) that didn’t exist back then, so it’s more tuned to real 2020s problems instead of old issues.

How do you actually use these? (Link tag? Import? Inline?)

You just put the CSS reset at the very top of your stylesheet.

Options:

  • <link> to an external reset.css file
  • Or copy/paste it at the top of your main CSS file
  • Or @import it but that’s kind of discouraged now because it slows loading.

Most people just paste it directly into their CSS as the first thing.

What’s a CSS boilerplate? (vs. a reset)

A reset wipes everything clean.
A boilerplate adds a bunch of helpful starting styles like default font stacks, spacing rules, utility classes, styling for buttons, form elements, etc.
It’s basically a “starter kit” for your CSS, not just a cleanup.

Why use a boilerplate? (When does it help/hurt?)

It helps when you’re starting a project and want the basics already set up so you’re not reinventing boring stuff (like buttons or form spacing).
It hurts if you don’t need all of it, it can add a ton of styles you end up fighting against or deleting.

Why is Normalize.css different from resets? (Philosophy shift)

Normalize doesn’t wipe everything out.
Instead of clearing all defaults, it tries to make the browser defaults consistent. So instead of “zero everything,” it’s “keep the good defaults, fix the weird ones.”
It’s more gentle, more practical, and less destructive than the old-school resets.

What changed between 2007 and now? (From Meyer to modern resets)

Browsers are way more consistent now, so we don’t need to nuke literally everything anymore.
Developers realized that destroying all default styles was actually a pain because you had to restyle basic elements from scratch.
Modern resets focus on small, targeted fixes things browsers still disagree on (like images, buttons, text wrapping), not the giant universal wipeout.

What browser inconsistencies bothered developers in 2007?

Stuff like random margins on <body>, different default line-height, lists showing bullets differently, headings having different sizes in every browser, and form elements being totally inconsistent between browsers. Basically everything looked unpredictable.

Other reset/normalize approaches (2–3 examples)

  • Normalize.css: The famous “don’t wipe everything, just fix inconsistencies” approach.
  • Sanitize.css: A modern, more opinionated normalizer.
  • Andy Bell’s Modern CSS Reset: Minimal and focused on practical issues.
